# Phase 2.5b — Versioning & rollback (spec)

**Status**: Spec, not yet built
**Owner**: Claude (tier3_owner: claude)
**Depends on**: Phase 2.5a (plans + plan_categories + PlanService, shipped 2026-04-19)
**Blocks**: Phase 2.5c (what-if scenarios), 2.5d (publish-to-Sheets)

## Goal

Every save of a plan category (create / update / delete / rename / importer
overwrite) leaves a restorable snapshot. The lost-edit window currently
exists: if the importer runs while an operator is mid-edit, the operator's
work is silently archived with no restore path. 2.5b closes that window by
persisting the pre-change snapshot and surfacing a per-plan history +
restore UI. It also widens `plans.status` to admit `draft` and `scenario`
values, which 2.5c needs.

Scope is deliberately bounded: this phase adds history, not branching. The
operator can see what changed and roll back, but cannot yet fork the plan
or A/B compare scenarios — that's 2.5c.

## In scope

- New SQLite table `plan_revisions` (one row per save event).
- `PlanService` instruments every write to create a revision before
  mutating `plan_categories`.
- Importer (`BudgetImporter.import_budget`) creates a revision with
  `source='importer'` on the about-to-be-archived active plan, so
  "importer overwrote my edits" is always recoverable.
- `GET /ui-api/plan/revisions` list + `GET /ui-api/plan/revisions/<id>`
  detail + `POST /ui-api/plan/revisions/<id>/restore` endpoints (plus
  `/api/*` mirrors).
- `plans.status` CHECK rebuilt to include `draft` and `scenario` via
  SQLite shadow-table migration inside `Database.initialize()`.
- `/planning` page grows a **History** button opening a right-side
  `<Sheet>` with per-revision rows, diff preview, and Restore.
- Retention policy applied on every insert: keep the last 50 revisions
  per plan OR anything newer than 7 days, whichever is larger.
- Tests (pytest + vitest) covering all of the above.

## Out of scope

- Branching / scenarios (2.5c).
- Revision annotations / tagging by the operator (nice-to-have, not
  load-bearing for rollback).
- Cross-plan history (revisions are scoped to a single `plan_id`).
- Publishing a revision back to Google Sheets — that's 2.5d.

## Schema

```sql
CREATE TABLE plan_revisions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    plan_id         INTEGER NOT NULL REFERENCES plans(id) ON DELETE CASCADE,
    created_at      TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    source          TEXT    NOT NULL CHECK(source IN (
                        'ui_create','ui_update','ui_delete','ui_rename',
                        'importer','migration','restore'
                    )),
    summary         TEXT,                 -- UI-provided (e.g. "Bills › Rent $1,200 → $1,300")
    change_count    INTEGER NOT NULL,     -- denormalized for list rendering
    snapshot_json   TEXT    NOT NULL      -- full plan_categories snapshot as JSON
);

CREATE INDEX idx_plan_revisions_plan_created
    ON plan_revisions (plan_id, created_at DESC);
```

`snapshot_json` is the complete plan_categories set at revision time
(post-change for ui_* sources, pre-archive for the importer source). A
restore loads it, `DELETE`s the current rows, and re-inserts the
snapshot under the same `plan_id`. Rows reuse their original integer IDs
only if they still exist; otherwise the restore assigns fresh IDs
(AUTOINCREMENT).

**Why JSON blob over a normalized child table**: the full snapshot is
small (five blocks × dozens of categories). Storing one blob keeps the
restore atomic — no stitching across two tables — and keeps the list UI
cheap (no `GROUP BY` on a changes child table). `change_count` +
`summary` give the list enough information to render without parsing
JSON.

## plans.status CHECK migration

Current `plans.status` CHECK (from Phase 2.5a): `CHECK(status IN
('active','archived'))`. Widen to `CHECK(status IN
('active','archived','draft','scenario'))`.

SQLite doesn't support `ALTER TABLE ... ADD/DROP CHECK`. The shadow-
table dance:

1. `CREATE TABLE plans_new (... CHECK(status IN ('active','archived','draft','scenario')) ...)`
   — mirror every other column and constraint from current `plans`.
2. `INSERT INTO plans_new SELECT * FROM plans;` — copy rows.
3. `DROP TABLE plans;`
4. `ALTER TABLE plans_new RENAME TO plans;`
5. Recreate `idx_plans_active_per_year` (the partial unique index).
6. `PRAGMA foreign_keys = ON;` stays pinned; `plan_categories.plan_id`
   FK still resolves by name after the rename.

Gated inside `Database.initialize()` by a `_migrate_plans_status_check()`
helper that reads `PRAGMA table_info(plans)` or inspects
`sqlite_master.sql` to detect the pre-migration schema. Idempotent —
running against the new shape is a no-op.

The migration must run inside the same `initialize()` transaction as
the existing 2.5a hydrations so partial upgrades never leave the schema
half-migrated.

## PlanService changes

Every write path gains a `revision_source` parameter (defaulted to the
appropriate value) and creates the `plan_revisions` row **before**
executing its own mutation, inside the same transaction:

```py
def update_category(
    self, category_id: int, plan_id: int, updates: dict[str, Any]
) -> PlanCategory:
    with self.db.transaction() as cur:
        self._snapshot_plan(cur, plan_id, source='ui_update', summary=...)
        # ... existing update logic ...
```

Each write method composes a human-readable `summary` from its inputs
(e.g. `"Bills › Rent planned $1,200.00 → $1,300.00"`). For deletes:
`"Removed Bills › Rent"`. For renames:
`"Renamed Bills › Rent → Bills › Rent (adjusted)"`.

`restore(revision_id)` is new:

```py
def restore(self, revision_id: int) -> PlanCategory | None:
    # Load revision, assert it belongs to a plan the caller can write.
    # Snapshot the current state with source='restore'.
    # DELETE FROM plan_categories WHERE plan_id = :plan_id.
    # INSERT FROM snapshot_json rows.
    # Bump plans.updated_at.
    # Returns the restored plan head for cache invalidation.
```

Retention: after inserting a revision, execute

```sql
DELETE FROM plan_revisions
WHERE plan_id = :plan_id
  AND id NOT IN (
    SELECT id FROM plan_revisions
    WHERE plan_id = :plan_id
      AND (created_at >= datetime('now','-7 days') OR id IN (
          SELECT id FROM plan_revisions
          WHERE plan_id = :plan_id
          ORDER BY id DESC LIMIT 50
      ))
  );
```

This keeps "last 50 rows OR anything in the last 7 days", whichever is
larger. Cheap, deterministic, no background job.

## Importer race-window

`BudgetImporter.import_budget()` already archives the prior active plan
for the year before inserting the new one. Add a snapshot immediately
before the archive, tagged `source='importer'` with summary
`"Importer overwrote plan on {YYYY-MM-DD HH:MM}"`. This does not block
the overwrite — the importer is still the source of truth for the
baseline — but it makes the operator's in-flight edits recoverable via
the standard history + restore UI with zero new import flags.

## API

All four mirrored on `/api/*` and `/ui-api/*`; UI writes require
`X-Finclaide-UI: 1` as usual.

| Method | Path                                            | Response                                     |
|--------|-------------------------------------------------|----------------------------------------------|
| GET    | `/plan/revisions?plan_id={id}&limit=50`         | `{ revisions: [ {id, plan_id, source, summary, change_count, created_at} ] }` |
| GET    | `/plan/revisions/{id}`                          | `{ id, plan_id, source, summary, change_count, created_at, snapshot: [PlanCategory, ...] }` |
| POST   | `/plan/revisions/{id}/restore`                  | `{ plan: ActivePlanResponse }` — same shape as `GET /plan/active`, lets the client swap in the response without a follow-up fetch |

Errors: 404 when revision / plan missing; 409 when `plans.busy` (e.g.
import running) to avoid stacking a restore over an in-progress
import.

## UI

`/planning` header grows a **History** button next to **Add row**,
disabled while `status.busy` is truthy. Clicking it opens a right-side
`<Sheet>` (same primitive as the edit Sheet) with:

- List of revisions, newest first. Each row shows source badge
  (color-coded: `ui_*` neutral, `importer` amber, `restore` slate),
  relative time ("3 min ago"), summary line, change count.
- Clicking a row reveals an inline diff: block-by-block list of
  categories whose planned or annual-target amount differs from the
  current plan, with a colored delta. No full JSON dump.
- "Restore this revision" button on the detail; confirms via a
  secondary `Dialog` ("Restore to this revision? Current state will be
  saved in a new revision first.") because restores are invasive.
- After restore: invalidate `['plan']` + `['summary']` + reactivate the
  block tab the restored plan lands on.

Where this disagrees with the existing React a11y surface: the History
button must be keyboard-reachable from the planning-page header nav
order. Covered in the test plan.

Three commits, one per build slice, verified at each boundary:

1. **Slice 1 — Schema + PlanService + tests.** `plan_revisions` table,
   `plans.status` CHECK migration, PlanService write paths instrument,
   retention prune, backend tests. No API, no UI. `pytest` full suite

## Open questions / deferred

- **Multi-operator concurrency.** Current app is single-household, so
  two simultaneous saves aren't expected. If two browsers save at once,
  each produces its own revision — last-write-wins on
  `plan_categories` but neither save is lost from history. Acceptable
  for now; revisit if 2.5c introduces per-operator scenarios.
- **Revision diff granularity.** Spec shows a category-level diff
  derived client-side from `snapshot_json` vs current plan. If the diff
  gets expensive at scale, move it server-side as a computed view.
- **Exporting history.** Not in 2.5b. If useful, a follow-up phase can
  add `GET /plan/revisions/export` as `.ndjson` for offline audit.
