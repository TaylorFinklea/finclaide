# Phase 2.5c — What-if Scenarios

## Context

Phase 2.5b shipped per-plan revision history (`plan_revisions` table,
PlanService instrumentation, importer pre-archive snapshot, History
UI with diff + restore). The schema CHECK on `plans.status` already
admits `'scenario'`, and the migration view-drop bug surfaced during
smoke is fixed. With that foundation, we have everything needed to
introduce scenario plans without further plumbing on the audit/undo
side.

Phase 2.5c delivers the operator workflow: branch off the active
plan, edit freely, project hypothetical changes, compare against
actuals, then commit or discard. The roadmap framing is
**"Branch off active plan; edit; compare projected variance vs
actuals; commit or discard."**

The mental model is a four-state cycle. **Active** is the live plan
(source of truth, single row). **Sandbox** is one transient unnamed
scenario row used for "edit and try"; at most one exists at any
moment. **Saved** scenarios are named, persistent rows used for
named alternatives ("Summer budget", "If Daisy stops working").
**Projection** is stateless slider exploration on `/scenarios`;
no DB row, output is an annual delta + per-category preview, and
its results can be applied into a Sandbox.

Transitions: Active→Projection, Active→Sandbox, Projection→Sandbox,
Sandbox→Saved, Sandbox→Active (commit), Sandbox→discard,
Saved→Sandbox (fork — Saved is never mutated in place),
Saved→Active (commit), Saved→delete.

Locked design choices from brainstorming:

- One sandbox max. Starting another auto-parks the existing one as
  a Saved scenario (modal prompts for a name, defaults
  `"Untitled scenario, <today>"`).
- Compare-against-actuals is available on both Sandbox and Saved.
- Editing a Saved scenario forks into a Sandbox; the Saved row is
  never mutated.
- UI surface: `/planning` gets a "Try a what-if" button that drops
  into Sandbox mode in place. `/scenarios` is the management route
  (Saved list + Projection panel).
- Projection axes: per-category percent sliders + a "+ add
  hypothetical line ($/mo)" affordance.
- Comparison view: per-category drilldown with 6-month sparklines.
- Commit drops the scenario name (plans don't have user-facing
  names except for scenarios). Attribution lives in the
  `plan_revisions` summary on the archived prior-active row.

## Schema delta

Single column + two partial unique indexes. The existing `plans.name`
is `NOT NULL` and holds the sheet/budget title (e.g. `"2026 Budget"`),
so we add a separate nullable `label` column for the user-facing
scenario name.

```sql
ALTER TABLE plans ADD COLUMN label TEXT;

CREATE UNIQUE INDEX IF NOT EXISTS idx_plans_one_sandbox
  ON plans(status) WHERE status='scenario' AND label IS NULL;

CREATE UNIQUE INDEX IF NOT EXISTS idx_plans_saved_label_unique
  ON plans(label) WHERE status='scenario' AND label IS NOT NULL;
```

`plans.label` is `NULL` for `active` / `archived` / unnamed Sandbox
rows; it holds the user-provided string for Saved scenarios. The
first index enforces the "at most one Sandbox" invariant — among
rows where `status='scenario' AND label IS NULL`, the indexed
`status` value (always the constant `'scenario'`) must be unique,
so only one row can satisfy. The second index forbids duplicate
Saved names.

For scenario rows, `plans.name` inherits the source plan's name
(e.g. the active plan's `"2026 Budget"`) — keeps the year/source
context attached. The user-visible scenario name is `label`.

`SCHEMA_SQL` in `src/finclaide/database.py` is updated so a fresh
install matches an upgraded one. The migration is idempotent: it
checks `PRAGMA table_info(plans)` for an existing `label` column
and exits if present.

## API surface

All under `/ui-api/scenarios/...`. Existing
`PATCH /ui-api/plans/{id}/categories/{cat_id}` is reused for
editing scenario categories — scenarios are just plans with a
different `status`.

| Method/Path | Purpose |
|---|---|
| `POST /ui-api/scenarios` | Body: `{from: "active" \| {scenario_id}, name?}`. Deep-copies `plan_categories` into a new scenario row. `name` omitted → Sandbox; `name` set → Saved. If a Sandbox exists and the new one would also be a Sandbox, frontend has already auto-parked it via `POST .../save`. |
| `GET /ui-api/scenarios` | Lists `status='scenario'` rows: id, name, `created_at`, `updated_at`, change_count vs active. |
| `GET /ui-api/scenarios/{id}` | Full scenario plan + categories (same shape as `GET /ui-api/plans/{id}`). |
| `POST /ui-api/scenarios/{id}/save` | Body: `{name}`. Promotes Sandbox → Saved. Validates uniqueness. |
| `POST /ui-api/scenarios/{id}/fork` | Auto-parks any existing Sandbox (frontend prompts first), creates a new Sandbox seeded from this scenario. |
| `POST /ui-api/scenarios/{id}/commit` | Snapshots current Active to `plan_revisions` (source `commit_replaced_active`, summary `"Replaced by scenario: <name or 'Sandbox'>"`), flips this scenario to `status='active'`, clears `name`, archives the prior active row. Wrapped in `operation_lock.guard("plan_commit")`. |
| `DELETE /ui-api/scenarios/{id}` | Snapshots into `plan_revisions` (source `scenario_discarded`), then deletes. Recoverable from History. |
| `POST /ui-api/scenarios/compare` | Body: `{scenario_id?, projection?: {axes}}`. Returns `{rows: [{category_id, name, block, group, planned_active, planned_scenario, actuals_avg_6mo, variance, sparkline:[m1..m6]}], totals}`. Powers Sandbox/Saved drilldown and Projection preview. |
| `POST /ui-api/scenarios/projection/apply-to-sandbox` | Body: `{axes}`. Materializes Projection into Sandbox edits — creates the Sandbox if absent, applies category-level deltas, returns sandbox id. |

No private `/api/*` scenario endpoints in v1 — scenarios are a UI
concept; external AI tools continue to operate on the active plan
only. Auth: all endpoints under `/ui-api/`, cookie session, same as
existing plan editing.

Errors: 409 from `operation_lock.guard` during import/sync/
reconcile/commit; 404 for missing scenario; 422 for name collision
on save; 409 from the partial unique index if frontend tries to
create a second Sandbox without auto-park.

## Data flow & service layer

`PlanService` extended (or a sibling `ScenarioService` introduced —
TBD during slice 1 build) with:

- `create_scenario(from, name=None) -> plan_id` — copies
  `plan_categories` rows, writes a `plan_revisions` row tagged
  `source='scenario_created'` for the new scenario id.
- `save_scenario(scenario_id, name)` — sets `plans.name`,
  validates uniqueness.
- `fork_scenario(saved_id) -> sandbox_id` — convenience that
  wraps `create_scenario(from=saved_id)`.
- `commit_scenario(scenario_id)` — atomic: snapshot active,
  archive active, flip scenario→active+clear name. Single SQL
  transaction; reuses the same archival path as the importer.
- `discard_scenario(scenario_id)` — snapshot+delete.
- `compare(scenario_id_or_projection)` — pure read, joins
  `plan_categories` against `transactions` rolled up per category
  per month over the last 6 months. Sparkline values are
  milliunits; frontend handles formatting.

Existing `PATCH /ui-api/plans/{id}/categories/{cat_id}` handler
already creates `plan_revisions` rows on every write — that
behavior is correct for scenarios too. `source` stays
`ui_update`; the row's `plan_id` distinguishes active from
scenario in History.

Sandbox edits between create and commit/discard generate revision
rows just like active-plan edits do. Retention prune (50 OR 7d)
already runs per-plan on each write; no extra plumbing.

## UI

### `/planning` (additions only)

Header gains **[Try a what-if]** next to **[History]**. Click:

- No sandbox exists → create one, enter Sandbox mode in place.
- Sandbox exists → banner reads "Continuing your sandbox from
  <relative time>" with a "Start fresh" link. Start fresh opens
  the auto-park modal (name field, default
  `"Untitled scenario, Apr 26"`), then creates a new sandbox.

Sandbox-mode banner:

```
⚠ Sandbox · unsaved · 4 changes  [Compare] [Save] [Discard] [Commit]
```

Edits use the existing PATCH wiring; only the target plan id
changes. Save opens a name modal. Commit opens a confirmation
modal showing change count + warning that the active plan will be
archived to History (where it can be restored). Discard opens a
confirmation modal.

### `/scenarios` (new route)

Two stacked sections:

1. **Saved scenarios** — list with name, last edited, and per-row
   actions: **Open** (forks to Sandbox, routes to `/planning`),
   **Compare** (opens drawer), **Make active** (commit
   confirmation), **Delete** (confirmation).
2. **Pure projection** — N rows of `[category picker] [percent
   slider]` plus a "+ add hypothetical line ($/mo)" row. Live
   preview card below: annual delta vs active, against last 6mo
   actuals. **[Apply to Sandbox]** button materializes the slider
   state into a Sandbox and routes to `/planning`.

### Comparison drawer (shared)

Slide-in drawer (or full-page if it grows past comfortable drawer
size during build). Layout:

```
Headline: "This scenario would have ended +$2,140 ahead vs active
over 6mo."

Category | Active | Scenario | Actuals avg | Variance | 6mo
(sortable by Variance desc; sparkline = 6-bucket inline SVG)
```

Sparkline: 6 monthly buckets per category, normalized per-row,
grayscale. No charting library needed — inline SVG `<polyline>`
or `<rect>` bars, ~30 LOC.

### Components

Reused: planning table cells, plan-edit PATCH wiring, `bits-ui`
Dialog/Sheet (in use on History), History route layout patterns.

New: Sandbox-mode banner, scenario list card, projection
sliders + add-line panel, sparkline cell, comparison drawer.

## Out of scope

- Multiple concurrent sandboxes (locked to one).
- Shared/team scenarios (single-household app, v1).
- Scenario-level access control (no ACL system in v1).
- Export of scenario diff (v1 keeps it in-app; CSV export is a
  later story).
- "Schedule" or "what-if-by-date" projections (only static
  axis-style projections in v1).
- Sandbox auto-resume across browser sessions (Sandbox persists
  in DB, so it does survive — but no special "you have an
  unsaved sandbox" toast; the `/planning` banner serves that
  role when you next visit).
